# Proxima API

FastAPI backend для маркетплейса цифровых услуг. Реализованы пользователи, услуги,
заказы и личные сообщения.

## Запуск

Из корня проекта:

```powershell
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements-dev.txt
alembic upgrade head
uvicorn app.main:app --reload
```

Также поддерживается прямой запуск:

```powershell
cd app
py main.py
```

Swagger: <http://127.0.0.1:8000/docs>.

После добавления новых таблиц обязательно выполните `alembic upgrade head`. Вторая
миграция создаёт сущности маркетплейса и добавляет поля профиля.

## Основные сущности

- `User` — аккаунт, публичное описание и URL аватара.
- `Listing` — услуга продавца, цена хранится целым числом в минимальных единицах валюты.
- `Order` — заказ со снимком названия и цены услуги на момент покупки.
- `Conversation` — прямой диалог либо уникальный диалог конкретного заказа.
- `Message` — сообщение с редактированием, мягким удалением и отметкой прочтения.

## Авторизация

Для защищённых запросов передавайте access-токен:

```http
Authorization: Bearer ACCESS_TOKEN
```

| Метод | Endpoint | Назначение |
|---|---|---|
| POST | `/api/v1/auth/register` | Регистрация |
| POST | `/api/v1/auth/login` | Получение пары токенов |
| POST | `/api/v1/auth/refresh` | Обновление пары токенов |
| POST | `/api/v1/auth/logout` | Завершение refresh-сессии |

## Профили и аватары

| Метод | Endpoint | Назначение |
|---|---|---|
| GET | `/api/v1/users/me` | Свой полный профиль |
| PATCH | `/api/v1/users/me` | Изменить username |
| POST | `/api/v1/users/me/avatar` | Загрузить JPEG/PNG/WebP до 5 МБ |
| DELETE | `/api/v1/users/me/avatar` | Удалить аватар |
| GET | `/api/v1/users/{user_id}` | Публичный профиль без email |

Аватар отправляется как `multipart/form-data`, имя поля — `file`. Сервер приводит
изображение к WebP и возвращает путь вида `/media/avatars/...`.

## Услуги

| Метод | Endpoint | Назначение |
|---|---|---|
| GET | `/api/v1/listings` | Поиск и пагинация активных услуг |
| POST | `/api/v1/listings` | Создать услугу |
| GET | `/api/v1/listings/{id}` | Получить услугу |
| PATCH | `/api/v1/listings/{id}` | Изменить свою услугу |
| DELETE | `/api/v1/listings/{id}` | Отключить свою услугу |

Фильтры списка: `q`, `category`, `seller_id`, `min_price`, `max_price`, `offset`, `limit`.

Пример создания:

```json
{
  "title": "Прокачка персонажа",
  "description": "Подниму персонажа до указанного уровня",
  "category": "games",
  "price_minor": 150000,
  "currency": "RUB",
  "delivery_days": 3
}
```

## Заказы

| Метод | Endpoint | Назначение |
|---|---|---|
| GET | `/api/v1/orders` | Свои покупки и продажи |
| POST | `/api/v1/orders` | Заказать активную услугу |
| GET | `/api/v1/orders/{id}` | Получить свой заказ |
| PATCH | `/api/v1/orders/{id}` | Покупателю изменить требования |
| POST | `/api/v1/orders/{id}/status` | Изменить статус |
| DELETE | `/api/v1/orders/{id}` | Отменить ожидающий либо скрыть конечный заказ |

При создании заказа клиент передаёт только `listing_id` и `requirements`. Продавец,
название, цена и валюта берутся на сервере из услуги.

```json
{
  "listing_id": "UUID",
  "requirements": "Нужно получить 50 уровень"
}
```

Редактирование и смена статуса требуют актуальный `version`. После каждого изменения
сервер увеличивает версию. Если два клиента меняют один заказ одновременно, устаревший
запрос получает `409`.

Статусный сценарий:

```text
pending -> accepted -> in_progress -> delivered -> completed
   |           |             |             |
cancelled   cancelled     disputed      disputed
```

- продавец принимает заказ, начинает работу и отмечает доставку;
- покупатель завершает доставленный заказ;
- ожидающий заказ можно отменить;
- спор можно открыть после принятия заказа;
- покупатель может закрыть спор принятием результата, продавец — отменой заказа;
- удаление не уничтожает общую историю: заказ скрывается только у вызвавшего участника.

## Чаты

| Метод | Endpoint | Назначение |
|---|---|---|
| GET | `/api/v1/chats` | Список диалогов, unread count и последнее сообщение |
| POST | `/api/v1/chats` | Открыть прямой диалог или получить чат заказа |
| GET | `/api/v1/chats/{id}` | Получить диалог |
| GET | `/api/v1/chats/{id}/messages` | Сообщения с пагинацией |
| POST | `/api/v1/chats/{id}/messages` | Отправить сообщение |
| PATCH | `/api/v1/chats/{id}/messages/{message_id}` | Изменить своё сообщение |
| DELETE | `/api/v1/chats/{id}/messages/{message_id}` | Удалить своё сообщение |
| POST | `/api/v1/chats/{id}/read` | Отметить чужие сообщения прочитанными |

Прямой диалог:

```json
{"recipient_id": "UUID"}
```

Чат заказа создаётся автоматически вместе с заказом. Получить его можно повторяемым
запросом:

```json
{"order_id": "UUID"}
```

Повторное создание прямого диалога между той же парой пользователей возвращает уже
существующий диалог. Посторонний пользователь не может читать сообщения заказа.

`GET /messages` возвращает новые сообщения первыми. Для следующей страницы передайте
`before_id` последнего сообщения текущей страницы.

## Тесты и ревью

```powershell
python -m pytest -q
ruff check .
ruff format --check .
```

Тесты используют временную SQLite-базу и не изменяют PostgreSQL из `.env`. Они проверяют
авторизацию, профиль, обработку аватара, CRUD услуг, права участников заказа, полный
статусный цикл, версионные конфликты, заказной чат, прямой чат и сообщения.

## Деплой на Vercel

Проект распознаётся Vercel как FastAPI-приложение через `app.main:app`. В production
обязательно нужна внешняя PostgreSQL; файловая SQLite используется только локально.

```powershell
vercel link
vercel integration add neon --plan free_v3 -m region=fra1 -m auth=false
alembic upgrade head
vercel --prod
```

Интеграция Neon автоматически добавляет `DATABASE_URL` и `DATABASE_URL_UNPOOLED` в проект.
Приложение предпочитает unpooled URL и автоматически преобразует обычный PostgreSQL URL
в формат асинхронного драйвера SQLAlchemy. Аватары хранятся в таблице `avatars`, поэтому
не зависят от временной файловой системы Vercel Functions.
